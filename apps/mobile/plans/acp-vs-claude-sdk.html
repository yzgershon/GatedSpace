ACP-aligned protocol vs Claude SDK harness

Both trees implement the same architecture — a seq-stamped envelope journal on the host, one WebSocket replayed with ?since=, and a pure client-side fold into timeline items. What differs is the vocabulary riding inside the envelope, and who does the interpretive work. UI code is excluded from every number below.

1Where they actually differ

Seven divergences (D1–D7) carry the wire-protocol signal; everything else in phases ①–⑨ is the same machine twice. Ranked, roughly, by how much they should weigh on the decision. D8–D10 are a different species: not “who does the work” but “can the vocabulary even say it” — covered in phases ⑩–⑮ and the §6 fidelity matrix.

D1 — Where history lives hops ↓

ACP: host SQLite keeps the public session → native ACP session binding. A restarted host lists the row as offline and resurrects it on demand with session/load. The adapter replays its native on-disk transcript into a fresh, bounded 5,000-frame journal.

SDK: the journal is only a delivery buffer; truth is Claude's native JSONL transcript on disk — survives restarts, powers retry() epochs.

Both sides recover completed history from Claude's native store. The ACP host owns registry metadata, not a durable transcript. getMessages still pages the bounded ring, so content older than that retained window is unavailable; disk-backed paging is required follow-up work. In-flight turns die with the host on both sides.

D2 — Stream vocabulary hops ↓

ACP: streams too — one agent_message_chunk per text delta, live during the turn; chunks accumulate into one message and are never replaced (the adapter diffs the SDK's consolidated message against what already streamed and forwards only any un-streamed tail). Tool calls arrive as UI-ready tool_call frames (title/kind/status on the wire).

SDK: journals the raw model stream verbatim (message_start, content_block_delta, …) and then a complete assistant message that supersedes the accumulated deltas at fold time — ~10× the journal rows, interpretation deferred to the client fold.

Both sides stream token-by-token. The divergence is the vocabulary (semantic chunks vs raw stream events) and whether a later authoritative message replaces what streamed — ACP: no; SDK: yes.

D3 — Permission contract hops ↓

ACP: ships the options on the wire; the client taps an optionId back — zero decision logic on the device.

SDK: ships raw input + rule suggestions; the client must build a full PermissionResult and invent the Allow / Always / Deny UX.

D4 — Questions (AskUserQuestion) hops ↓

ACP: the host converts the elicitation form into ordinary permission cards — mobile has zero question code.

SDK: the host forwards the raw tool input and mobile special-cases it — the host has zero question code; carries 3 parallel interaction channels vs ACP's 1.

Same tool, opposite ends do the work.

D5 — Process model hops ↓

ACP: boots a child process per session and needs 3 stdio round-trips (initialize → session/new → set_mode) before it's usable.

SDK: one in-process query() call whose options carry cwd, mode and model; its callbacks never cross a process boundary.

D6 — Sending a prompt hops ↓

ACP: takes plain ContentBlocks, allows mid-turn sends, and the host echoes your bubble into the journal instantly.

SDK: needs a client-built SDKUserMessage, only accepts when idle, and your bubble waits for the CLI's echo.

D7 — Turn end hops ↓

ACP: turn-end is host-internal — a stopReason answering a parked RPC; devices only ever see the state snapshot.

SDK: journals an explicit result frame with duration, cost and usage — billing telemetry for free.

D8 — Subagents hops ↓

ACP: the spawn becomes a generic think tool call, subagent prose is deliberately dropped, and the tree survives only via the non-standard _meta.claudeCode.parentToolUseId our fold happens to know about.

SDK: a typed spawn (subagent_type, model, isolation), every subagent message tagged with parent_tool_use_id, transcript readable later by (sessionId, agentId).

D9 — Task graph → flat checklist hops ↓

ACP: the four Task tools are suppressed; each mutation re-emits one flat plan — content + status only, priority invented as "medium", everything else dropped on the adapter floor.

SDK: ID-addressed tasks with dependencies (blocks/blockedBy), owner, metadata — a queryable graph.

D10 — Workflows & goals don't cross hops ↓

ACP: the workflow crosses as one opaque kind:"other" tool call; goals don't cross at all — the adapter has no mapping for them.

SDK: a first-class Workflow tool (resumable runs, taskId/runId, transcript dir) and an active_goal message type.

Same skeleton grew on both sides

Envelope {seq, sessionId, ts, frame}, append-only ring journal (cap 5,000), WS replay from a cursor, admission-ack sends, parked-promise permissions, full-snapshot state frames, pure fold reducer. The protocol choice didn't change the architecture — it changed the vocabulary, and who does the interpretive work.

ACP ships UI-ready events

Semantic update variants: tool_call / tool_call_update with patch-merge, titles and statuses born at the protocol layer, permission options carried on the wire, one interaction channel (AskUserQuestion mapped onto it via elicitation). Small client vocabulary; the fold does chunk-merge, patches, subagent tree.

SDK ships the raw model stream

Verbatim SDKMessages: partial deltas superseded by authoritative messages, uuid identity, tool results as user-role messages, three parallel interaction channels, client-built PermissionResult. Buys native-transcript history, retry epochs, catalog, usage/cost — costs +47% non-UI LOC and +49% ifs.

Devices — derived state only

envelopes[] (deduped by seq) → fold() → FoldedTimeline { items, meta, state, lastSeq }.

The thread renders from items; status chip, composer gate and pending cards render from state. Everything here is disposable — refolded from scratch on any resync.

Host — the only authority

SessionScopedState mutates only here; every mutation is journaled as a full-snapshot state frame — clients receive status, never compute it.

SessionJournal: per-session in-memory catch-up ring, gapless within one runtime incarnation, cap 5,000. Every append fans out to every subscribed socket in the same call.

Parked asks: ACP Map<requestId, resolve> of unanswered JSON-RPC requests · SDK three Deferred maps (permissions / dialogs / elicitations).

Agent — conversation truth

ACP: one claude-agent-acp child process per active session; its embedded SDK persists the native transcript. Host SQLite stores the lookup key, and session/load restores completed history.

SDK: in-process query() handle over a claude subprocess; history truth is Claude's native JSONL transcript on disk, which outlives the journal.

5Seq journal & UI folding — the lower-level contract

Where the two sides converged exactly, and where the SDK fold earns its extra 109 lines.

ACP

seq assignment
Host only, per-session, gapless from 1. Append and WS fan-out happen in the same call — no gap window.
row granularity
One row per semantic event: every chunk, every tool_call / patch, every permission edge, every state snapshot. No write-time coalescing.
append vs update
Rows are immutable — everything appends. All merging is client-side at fold time: chunk concat, patch merge, resolution attach.
history source
Claude's native store survives; the host reaches it only through full session/load replay. getMessages then walks the bounded ring backwards (cursor s<seq>; state/reset frames excluded), so this is not complete disk-backed pagination yet.
replay & dedupe
Server replays (since, latest]; client drops seq ≤ lastSeq, treats seq > lastSeq+1 as a gap (reconnect from cursor), treats reset as stop-and-resync. Backoff 250ms → 10s.
fold hot spots
Consecutive-chunk merge · ACP patch semantics · subagent tree re-homing via _meta.claudeCode.parentToolUseId · placeholder tool item when a permission's tool_call fell off the loaded page.

Claude SDK

seq assignment
Same — host, per-session, gapless from 1. Plus epochs: retry() creates a fresh journal restarting at 1 and pushes reset("session_restarted") to old subscribers.
row granularity
One row per SDKMessage — every stream delta is its own row — plus interaction lifecycle edges and state snapshots.
append vs update
Same append-only journal. The fold replaces partial messages with the authoritative assistant message and flips tool / dialog items in place.
history source
The journal is only a recovery buffer. Truth = Claude's native JSONL transcript; transcript rows have no seq (folded at seq 0, uuid-keyed so live re-delivery upserts instead of duplicating).
replay & dedupe
Same semantics, plus cursor_ahead detection (first replayed seq below the cursor after a restart) and zod validation of every frame — an invalid frame forces a reconnect.
fold hot spots
Delta accretion + authoritative replacement (uuid identity) · tool linkage via toolUseID with a placeholder when the permission wins the race · tool results arriving as user-role messages · transcript-vs-live merge at seq 0.

ACP itself has no standard model for any row below "plan": no workflow entity, no task graph, no subagent addressing — _meta is the protocol's sanctioned extension point, and both adapters lean on it in mutually incompatible dialects (_meta.claudeCode.parentToolUseId vs _meta.codex.goal). ACP has no model for the hard rows

Conversation, permissions, modes, plans: standardized. Workflows, task graphs, subagent identity: not in the protocol at all — every adapter invents a private _meta dialect or silently flattens. The spike doc said it early: “_meta fields are the sanctioned extension point on every type.”

Good chat plane, not a workflow plane

As a common chat/control plane ACP earns its keep — one client vocabulary, permission options on the wire, plan checklists for free. As a portable workflow/multi-agent model it isn't one: the richest capabilities of both agents either flatten (tasks → checklist), go opaque (workflows), or vanish (subagent prose, goals).

What a rich Superset UI would need

Either agent-specific _meta contracts (we already parse claudeCode.parentToolUseId; we'd add codex.goal, and re-do it per agent) or a Superset-owned normalized layer above the adapters — native SDK/app-server in, one internal schema out. The waterfall above is the inventory of what that layer must preserve.

Read the numbers with this in mind: the ACP tree has had more UI iteration (plus an already-deleted MCP-bridge detour); the SDK tree carries heavier supporting infrastructure — a 978-line relay E2E script, packaged-host validation, 7 Maestro flows — all excluded above. Both are dev-gated. The ACP implementation persists a SQLite session registry and resurrects offline sessions on demand through session/load; the direct design reads the same native transcript through SDK helpers.
